## Goal

Three improvements to Plan Mode:
1. Approve plans directly from the AI Plans screen (so a missed toast isn't fatal).
2. Bulk-delete failed / cancelled / completed plans, with rows removed from the database.
3. Make the AI Plans screen fully responsive on phones (no horizontal overflow).

## 1. Approve from the AI Plans screen

`AIPlansScreen` currently surfaces Running / Failed / Cancelled / Completed only. Add a new top section:

- **"Awaiting approval"** — plans with `status = 'proposed'` (and steps.length > 0).
- Tapping a row opens the existing `PlanApprovalDialog` (already handles approve / cancel / refused / failed states). Reuse it — no logic duplication.
- Track open dialog separately from `PlanDetailDialog` so each section routes to the right one:
  - `proposed` → `PlanApprovalDialog`
  - everything else → `PlanDetailDialog` (current behavior)
- Add a small yellow badge / "Tap to review" hint on proposed rows so it's obvious they need action.
- Also surface `composing` plans (read-only "Planning…" row) so the user sees in-flight work.

After approval the dialog already closes and toasts; the screen list will refresh via the existing `["plans"]` query invalidation.

## 2. Bulk delete completed / failed / cancelled plans

**Database (migration):**
- Add RLS `DELETE` policy on `public.plans`: `using (auth.uid() = user_id)`. Currently DELETE is blocked entirely.

**UI in `AIPlansScreen`:**
- Each terminal section (Failed, Cancelled, Completed) gets a "Clear all" button in its header that only renders when the section is non-empty.
- Confirm with a native `confirm()` (lightweight; matches the app's existing minimal style).
- Delete via `supabase.from('plans').delete().eq('user_id', userId).eq('status', <sectionStatus>)`, then invalidate `["plans"]` and `["plans_pending_count"]`.
- Also add a per-row swipe/long-press affordance? — Out of scope; sticking to bulk to keep scope tight. (Can add later if needed.)
- Toast success/failure via sonner.

Note: deleting a plan row only removes the plan record itself. Media assets generated by the plan are independent rows in `media_assets` and stay in the gallery — that's the right behavior (user's generated content should not vanish because they cleared the plan log).

## 3. Mobile-responsive layout

Current `AIPlansScreen` uses `fixed inset-0` which is fine, but:
- `PlanApprovalDialog` / `PlanDetailDialog` use `max-w-lg` with no width constraint on small screens. On mobile they overflow because `DialogContent` defaults to `w-full max-w-lg`, but the inner content (long `user_request`, long `error_message`, JSON `pre` blocks) forces horizontal scroll.

**Fixes:**
- `DialogContent` → add `w-[calc(100vw-2rem)] max-w-lg mx-4 sm:mx-auto` and `max-h-[85vh] overflow-y-auto` to both dialogs.
- Wrap long text containers with `break-words` / `overflow-wrap-anywhere`. Replace `whitespace-pre-wrap` plain text with `whitespace-pre-wrap break-words`.
- `<pre>` blocks in `PlanDetailDialog` already have `overflow-x-auto` — keep, but constrain with `max-w-full`.
- In `AIPlansScreen` rows: the `truncate` on the request line is good; verify the badge + chevron don't push width. Use `min-w-0` on flex children (already present) and confirm header section titles wrap.
- Header buttons row: ensure "Clear all" and "Close" don't wrap awkwardly — use `flex flex-wrap gap-2`.
